## 기본 대화 설계 방식

Copilot Studio에서 Agent의 대화는 **Topic 단위**로 설계됩니다. 하나의 Topic은 하나의 사용자 의도(Intent)를 처리하는 대화 흐름입니다.

### 대화 설계 원칙

1. **하나의 Topic = 하나의 업무 시나리오**
   - 예: "휴가 잔여일 조회", "회의실 예약", "IT 장애 신고"
   - 여러 업무를 하나의 Topic에 넣지 않습니다.

2. **트리거(Trigger) 문구 설계**
   - 사용자가 실제로 입력할 만한 다양한 표현을 5~10개 등록합니다.
   - 예: "휴가 며칠 남았어?", "연차 잔여일", "남은 휴가", "휴가 확인"

3. **노드 흐름 설계**
   - 질문 → 처리 → 응답의 흐름으로 구성합니다.
   - 분기가 필요하면 **조건 노드**를 활용합니다.

### 대화 흐름 설계 예시

```
[트리거] "회의실 예약"
    ↓
[질문] "어떤 날짜에 예약하시겠어요?" → 변수: date
    ↓
[질문] "몇 시부터 몇 시까지?" → 변수: startTime, endTime
    ↓
[질문] "인원은 몇 명인가요?" → 변수: headCount
    ↓
[조건] headCount > 10 → 대회의실 / headCount <= 10 → 소회의실
    ↓
[Action] 회의실 예약 API 호출
    ↓
[메시지] "예약이 완료되었습니다! 📅 {date} {startTime}~{endTime}"
```

<div class="info-box tip">
<strong>💡 실무 팁</strong>
대화 흐름을 설계할 때는 먼저 <strong>종이나 화이트보드에 순서도</strong>를 그려보세요. 노드가 10개를 넘어가면 Topic을 분리하거나 Action으로 로직을 외부화하는 것을 고려하세요.
</div>

## 변수, 상태 관리 개요

Copilot Studio에서 **변수(Variable)**는 대화 중에 수집·저장·전달되는 데이터를 담는 그릇입니다.

### 변수 유형

| 유형 | 범위 | 설명 |
|---|---|---|
| **Topic 변수** | 해당 Topic 내부 | Topic 내에서만 사용. 다른 Topic에서 접근 불가. |
| **전역 변수 (Global)** | Agent 전체 | 모든 Topic에서 공유. 세션이 유지되는 동안 값이 보존됨. |
| **시스템 변수** | 시스템 제공 | `System.User.DisplayName`, `System.Conversation.Id` 등 |

### 변수 사용 예

```
# 질문 노드에서 사용자 입력을 변수에 저장
질문: "사번을 입력해주세요"
→ 변수 저장: Topic.employeeId (텍스트 타입)

# 전역 변수로 승격하여 다른 Topic에서도 사용
Global.employeeId = Topic.employeeId
```

### 상태 관리 시 주의사항

- **세션 유지 시간**: 기본적으로 30분간 대화가 없으면 세션이 종료됩니다.
- **세션 종료 시**: 모든 Topic 변수와 전역 변수가 초기화됩니다.
- **영속적 저장 필요 시**: Dataverse 또는 외부 DB에 Action을 통해 저장해야 합니다.

<div class="info-box warning">
<strong>⚠️ 주의</strong>
전역 변수를 과도하게 사용하면 흐름 추적이 어려워집니다. 전역 변수는 <strong>사용자 식별 정보, 세션 상태 플래그</strong> 등 꼭 필요한 경우에만 사용하세요.
</div>

## Action 기반 처리 흐름

**Action**은 Agent가 외부 시스템과 상호작용하는 핵심 메커니즘입니다. Topic의 대화 흐름 중간에 Action을 호출하여 데이터를 조회하거나 업무를 처리합니다.

### Action 호출 패턴

#### 패턴 1: Topic 내에서 직접 호출

대화 흐름의 특정 노드에서 Action을 직접 호출합니다.

```
질문 노드 → 변수 수집 → Action 호출 → 결과를 변수에 저장 → 응답
```

#### 패턴 2: Generative Orchestration

Agent의 Instructions에 Action 설명을 기술하면, LLM이 사용자 의도를 파악하여 **자동으로 적절한 Action을 선택·호출**합니다.

```
사용자: "내일 서울 날씨 어때?"
→ LLM 판단: 날씨 조회 Action 필요
→ Action 자동 호출 (파라미터: 서울, 내일)
→ 결과 기반 답변 생성
```

### Action 설계 모범 사례

| 원칙 | 설명 |
|---|---|
| **단일 책임** | 하나의 Action은 하나의 작업만 수행 |
| **명확한 입출력** | 입력 파라미터와 출력 값을 명확히 정의 |
| **에러 처리** | Action 실패 시 사용자에게 알릴 대체 흐름 준비 |
| **타임아웃 고려** | 외부 API 호출 시 응답 지연 가능성 대비 |

### Action 흐름 다이어그램

```
┌──────────────┐
│  사용자 입력   │
└──────┬───────┘
       ↓
┌──────────────┐     ┌─────────────────┐
│  Topic 매칭   │────→│  Generative     │
│  (트리거)     │     │  Orchestration  │
└──────┬───────┘     └────────┬────────┘
       ↓                      ↓
┌──────────────┐     ┌─────────────────┐
│  변수 수집    │     │  LLM이 Action   │
│  (질문 노드)  │     │  자동 선택      │
└──────┬───────┘     └────────┬────────┘
       ↓                      ↓
┌──────────────────────────────────────┐
│         Action 실행                   │
│  (Power Automate / 커넥터 / HTTP)    │
└──────────────┬───────────────────────┘
               ↓
┌──────────────────────────────────────┐
│         결과 반환 → 응답 생성          │
└──────────────────────────────────────┘
```

<div class="info-box tip">
<strong>💡 실무 팁</strong>
"반개발형"이라 부르는 이유는, 대화 설계와 변수 관리는 Copilot Studio UI에서 로우코드로 처리하되, 실제 비즈니스 로직은 <strong>Power Automate 플로우나 HTTP API</strong>로 분리하여 "개발 영역"과 "설정 영역"을 적절히 혼합하기 때문입니다.
</div>
